# Business Dashboard — Техническое задание

## Контекст

Дашборд для digital-агентства (ИП Громов А.В.) — учёт входящих оплат от клиентов,
привязка к проектам и статус закрывающих документов (актов).

Данные берутся из банковской выписки (PDF). Руководитель видит: кто заплатил,
за что, отправлен ли акт, подписан ли, где висят долги по документам.

---

## Стек

- **Backend**: Python 3.12, FastAPI + SQLAlchemy 2.0 (async) + Alembic
- **Database**: PostgreSQL 16
- **Frontend**: Node 20, Vue.js 3 + Vite + vue-router + axios
- **Docker**: docker-compose поднимает всё одной командой (app + postgres + frontend)
- **Тесты**: pytest + pytest-asyncio + httpx (TestClient) на отдельной БД `dashboard_test` на том же Postgres-сервисе (создаётся/мигрируется один раз за сессию, каждый тест в SAVEPOINT-транзакции с откатом)

---

## Запуск

Локально всё крутится только через Docker — никаких "поставь Python 3.12 и Node 20
себе на машину". Один `docker compose up --build` поднимает три сервиса:

| Сервис | Порт | Что внутри |
|---|---|---|
| `postgres` | 5432 | PostgreSQL 16 с volume для данных |
| `backend` | 8000 | FastAPI + uvicorn, авто-reload в dev, alembic upgrade head на старте |
| `frontend` | 5173 | Vite dev-server с прокси `/api → backend:8000` |

Тесты — `docker compose run --rm backend pytest`. Сами тесты используют отдельную
БД `dashboard_test` на том же `postgres` сервисе: при старте сессии она дропается,
создаётся заново и мигрируется; каждый тест работает в SAVEPOINT-транзакции,
которая откатывается на teardown — полная изоляция без truncate'ов.

Сиды накатываются командой `docker compose run --rm backend python -m app.seed`.

## Логика статусов акта

```
is_sent = true,  is_signed = true                              → closed
не closed + прошло 30+ дней с payment_date                     → attention
is_sent = true,  is_signed = false                             → waiting_signature
is_sent = false, is_signed = false                             → not_sent
```

Правила применяются сверху вниз: первое подходящее выигрывает. `attention` срабатывает
и для `not_sent`, и для `waiting_signature`, если оплата старше 30 дней — менеджеру
одинаково важно тормошить клиента в обоих случаях.

Статус пересчитывается на бэке при каждом обновлении акта и при запросе.

---

## Статус проекта (агрегированный)

- Все акты `closed` → проект `completed`
- Хотя бы один акт `attention` → проект `attention`
- Иначе → `active`

`on_hold` автоматически не выставляется — только вручную менеджером
(временная заморозка работ по проекту). При наличии оплат пересчёт его не трогает.

## API

### Clients
```
GET    /api/clients          — список клиентов с агрегатами (сумма, кол-во проектов)
GET    /api/clients/{id}     — клиент + его проекты
```

### Projects
```
GET    /api/projects         — список проектов с агрегатами
GET    /api/projects/{id}    — проект + его оплаты
```

### Payments
```
GET    /api/payments         — список оплат с фильтрами
POST   /api/payments         — создать оплату вручную
```

Фильтры для GET /api/payments:
- `project_id`
- `client_id`
- `date_from`, `date_to`
- `act_status` (not_sent / waiting_signature / closed / attention)
- `service_type`
- `search` (по назначению платежа или имени клиента)

### Acts
```
PATCH  /api/acts/{id}        — обновить статус акта (is_sent, is_signed, manager_comment)
```

### Summary
```
GET    /api/summary          — сводка по всем данным
```

Ответ summary:
```json
{
  "total_amount": 1234567.00,
  "total_projects": 19,
  "total_payments": 25,
  "closed_amount": 800000.00,
  "open_amount": 434567.00,
  "acts_not_sent": 5,
  "acts_waiting_signature": 3
}
```

### Import
```
POST   /api/import/statement — загрузить PDF банковской выписки
```

Принимает PDF-файл (multipart/form-data). На бэке:
1. Извлекает текст через `pdfplumber`
2. Парсит регулярками: дата, сумма, ИНН плательщика, название, назначение, номер счёта
3. Фильтрует операции (см. правила ниже)
4. Маппит платежи на клиентов/проекты (см. правила ниже)
5. Возвращает структурированный JSON с найденными платежами для превью
6. После подтверждения пользователем — создаёт клиентов, проекты, платежи и акты в БД

**Правила фильтрации операций:**

Берём только кредит (входящие). Дополнительно исключаем:
- ИНН плательщика = ИНН владельца счёта (свои переводы, возврат депозита)
- Контрагент = обслуживающий банк (АО "ФИН-МОСТ БАНК" по ИНН 7801001407) —
  начисление процентов по депозиту, комиссии
- Назначение содержит ключевые слова: `депозит`, `проценты по депозит`, `возврат депозит`

Расходные операции (дебет) — зарплата, аренда, налоги, переводы себе на личный счёт —
не попадают по построению (берём только кредит).

**Правила маппинга платёж → клиент → проект:**

- Клиент — по ИНН плательщика (если есть — переиспользуем, иначе создаём)
- Проект — один на (клиент, тип услуги). Тип услуги извлекается из назначения
  по словарю ключевых слов: `SEO`, `Директ`, `контекст`, `SERM`, `SMM`,
  `разработк`, `дизайн`, `лендинг`, `сопровожден`, `маркетинг`, `публикац`,
  `объявлен`, `тексты/копирайт`, `лицензи`, `презентац`, иначе `прочее`.
- Если такого проекта у клиента ещё нет — создаём с именем `{client.name} — {service_type}`
- Каждая строка выписки = одна `Payment` + автосоздаётся `Act` со статусом `not_sent`
- Если выписка явно перечисляет несколько счетов в одном платеже
  (например, "по счетам № 738, 791 и 792") — создаём ОДИН Payment с общей суммой,
  все номера складываем в `invoice_number` через запятую

Парсер заточен под формат АО "ФИН-МОСТ БАНК". Поддержка других банков и универсальный
LLM-парсинг — возможное улучшение в будущем (описано в README).

## Тесты

Стек: `pytest` + `pytest-asyncio` + `httpx`. Реальный Postgres (отдельная БД
`dashboard_test` на сервисе compose, SAVEPOINT-rollback на тест).

Что покрываем обязательно:

**Бизнес-логика:**
- Логика статусов акта (все 4 кейса)
- Агрегированный статус проекта
- Расчёт summary (суммы, счётчики)
- "Требует внимания" срабатывает через 30 дней

**API:**
- GET /api/payments — без фильтров, с каждым фильтром отдельно
- PATCH /api/acts/{id} — обновление is_sent, is_signed, статус пересчитывается
- GET /api/summary — корректность цифр
- POST /api/import/statement — на фикстурном PDF проверяем, что парсер возвращает корректную структуру и правильно отсекает свои/банковские/расходные операции

**Репозитории:**
- Фильтрация payments через реальную БД
- Агрегаты по клиентам и проектам

## Допущения

1. Авторизации нет — один пользователь (менеджер).
2. При импорте выписки берём только кредит, исключая свои переводы, банковские
   операции (проценты по депозиту, возврат депозита, комиссии) и расходы по построению.
3. Один акт на одну оплату (упрощение; в реальности акт может закрывать несколько оплат).
4. Один проект = (клиент, тип услуги). Несколько платежей одного клиента за одну услугу
   складываются в один проект как этапы.
5. Порог "требует внимания" — 30 дней с `payment_date`, вынесен в конфиг.
6. Парсер выписки — regex под формат АО "ФИН-МОСТ БАНК". LLM-парсинг (Claude Haiku /
   GPT-4o-mini) для произвольных форматов — возможное расширение, описано в README.
7. Фронт — Vue 3, простой интерфейс без дизайна, главное логика и фильтры.
